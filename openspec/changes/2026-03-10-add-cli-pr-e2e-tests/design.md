# Design: CLI PR E2E Tests

## Context

The CLI package has two distinct failure surfaces:

1. The CLI package itself must build correctly.
2. The project generated by the CLI must behave correctly after scaffolding.

The second surface is the more important one for end users and was previously unprotected in required CI. The implemented solution uses repository-local e2e runner scripts so that GitHub Actions and local verification execute the same flow.

## Goals / Non-Goals

### Goals

- Add a required PR gate for the real CLI scaffold path
- Pin the PR gate to Node `24.14.x`, matching generated project engine requirements
- Make failures diagnosable via persisted step logs
- Separate required PR coverage from broader nightly coverage
- Ensure the generated app passes `install`, `build`, and `type-check`

### Non-Goals

- Cross-platform matrix testing in PR
- Full nightly coverage as a required merge gate
- Reintroducing legacy `screenset create/copy` command coverage

## Decisions

### Decision 1: Separate workflow for CLI PR verification

**Decision**: Add `.github/workflows/cli-pr.yml` with a single required job `cli-pr-e2e`.

**Rationale**:
- Keeps CLI user-path verification independent from architecture validation
- Gives branch protection a stable, targeted check name
- Avoids overloading `main.yml` with long e2e concerns

### Decision 2: Node 24.14.x only for PR e2e

**Decision**: Run required CLI e2e only on `ubuntu-latest` + `Node 24.14.x`.

**Rationale**:
- Generated projects declare `engines.node >=24.14.0`
- The check should validate the minimum supported runtime for the scaffolded app
- A single-version PR gate reduces runtime and variance

### Decision 3: Two-tier verification

**Decision**:
- PR workflow runs the critical end-to-end scenario only
- Nightly/manual workflow runs broader scenarios (`--uikit none`, layer packages, migrate commands, invalid-name handling)

**Rationale**:
- Keeps PR latency acceptable
- Preserves broader regression coverage outside the merge-critical path

### Decision 4: Scripted e2e harness with artifacts

**Decision**: Store scenario logic in `packages/cli/scripts/e2e-*.mjs` instead of embedding shell scripts directly in workflow YAML.

**Rationale**:
- Reusable locally and in CI
- Easier to evolve than large inline shell blocks
- Allows step-level logging and structured summaries

## Required PR Flow

1. Build `@hai3/cli`
2. Create temp workspace
3. Run `hai3 create smoke-app --no-studio --uikit hai3`
4. Assert scaffolded files exist
5. Run `git init` in generated project
6. Run `npm install --no-audit --no-fund`
7. Run `npm run build`
8. Run `npm run type-check`
9. Run `hai3 validate components` on a clean project and expect success
10. Inject an invalid screen and expect `hai3 validate components` to fail
11. Run `hai3 scaffold layout -f`
12. Run `hai3 ai sync --tool all --diff`
13. Upload artifacts from the run

## Nightly Flow

The nightly/manual flow reuses the same harness patterns and additionally covers:

- app scaffold with `--uikit none`
- layer scaffolds: `sdk`, `framework`, `react`
- `hai3 migrate --list`
- `hai3 migrate --status`
- invalid package name rejection
- repeated `ai sync --diff` idempotency

## Template Hardening

The test gate is intentionally strict enough to force generated project correctness. If e2e uncovers scaffold regressions, template/runtime fixes are part of this change because the PR gate must reflect a genuinely working default scaffold.
